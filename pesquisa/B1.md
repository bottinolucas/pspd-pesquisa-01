# B.1 Conhecendo o gRPC

## Protobuf

O Protocol Buffers (protobuf) é um mecanismo de código aberto desenvolvido pela Google para serialização de dados estruturados. Seu principal objetivo é permitir a troca eficiente de informações entre aplicações, independentemente da linguagem de programação utilizada.

No protobuf, os dados são definidos em arquivos com extensão `.proto`, que funcionam como um contrato de comunicação entre cliente e servidor. Esses arquivos descrevem a estrutura das mensagens que serão transmitidas e podem ser utilizados para gerar automaticamente código em diversas linguagens, como Python, Java, C++, Go e C#.

Os dados são organizados em mensagens (messages). Cada mensagem representa um registro lógico de informações composto por uma série de campos. Cada campo possui um tipo, um nome e um identificador numérico único utilizado durante o processo de serialização.

### Exemplo utilizando JSON

```json
{
    "nome": "João",
    "idade": 22
}
```

### Exemplo utilizando protobuf

```proto
message Pessoa {
  string nome = 1;
  int32 idade = 2;
}
```

Nesse exemplo:

* `Pessoa` representa uma estrutura de dados;
* `nome` e `idade` são campos da mensagem;
* os números `1` e `2` são identificadores únicos utilizados internamente pelo protobuf durante a serialização.

Diferentemente de formatos textuais como JSON e XML, o protobuf converte os dados para um formato binário compacto antes da transmissão. Dessa forma, os nomes dos atributos não precisam ser enviados pela rede, reduzindo significativamente o tamanho das mensagens.

Durante a transmissão, o protobuf converte os dados para bytes binários, proporcionando:

* menor tamanho das mensagens;
* menor consumo de banda de rede;
* maior velocidade de serialização;
* maior velocidade de desserialização;
* melhor desempenho na comunicação entre aplicações.

Por essas características, o protobuf é utilizado como mecanismo padrão de serialização no gRPC, contribuindo para comunicações mais rápidas e eficientes em sistemas distribuídos e arquiteturas de microsserviços.

### HTTP/2 

O HTTP/2 é a segunda versão principal do protocolo HTTP e é utilizado pelo gRPC como camada de transporte para a comunicação entre aplicações. Diferentemente do HTTP/1.1, o HTTP/2 foi projetado para melhorar o desempenho da comunicação em rede, reduzindo a latência e aumentando a eficiência na troca de dados.

Uma das principais razões para o uso do HTTP/2 no gRPC é o suporte nativo a streams bidirecionais, permitindo que cliente e servidor enviem mensagens simultaneamente através da mesma conexão TCP. Essa característica é fundamental para implementar os diferentes modelos de comunicação oferecidos pelo gRPC, como server streaming, client streaming e bidirectional streaming.

Além disso, o HTTP/2 oferece recursos importantes:
- Multiplexação de streams: permite que múltiplas requisições e respostas sejam transmitidas simultaneamente por uma única conexão TCP.
- Compressão de cabeçalhos (HPACK): reduz a quantidade de dados transmitidos na rede.
- Conexões persistentes: elimina a necessidade de criar uma nova conexão para cada requisição.
- Comunicação full-duplex: cliente e servidor podem trocar mensagens ao mesmo tempo.

### Relação com o gRPC
Quando uma aplicação cliente realiza uma chamada gRPC, o fluxo ocorre da seguinte forma:
```
Cliente
   ↓
Mensagem protobuf
   ↓
gRPC
   ↓
HTTP/2
   ↓
Rede
   ↓
Servidor
```
Nesse processo:
- O cliente cria uma mensagem utilizando protobuf.
- O gRPC serializa a mensagem para formato binário.
- O HTTP/2 transporta essa mensagem pela rede.
- O servidor recebe os dados, desserializa a mensagem e processa a requisição.

Dessa forma, o protobuf é responsável por representar os dados, enquanto o HTTP/2 é responsável por transportá-los de forma eficiente, formando a base do funcionamento do gRPC.

### Unary calls

O Unary Call é o tipo mais simples de comunicação suportado pelo gRPC. Nesse modelo, o cliente envia uma única requisição ao servidor e recebe uma única resposta, de forma semelhante a uma chamada de função tradicional ou a uma requisição HTTP REST.
Esse modelo é indicado para operações em que apenas uma resposta é necessária, como consultar informações de um usuário, buscar um produto ou realizar um cálculo.
Funcionamento
```
Cliente                     Servidor
   |                             |
   |------ Requisição ---------->|
   |                             |
   |<------- Resposta -----------|
   |                             |
   ```
**1**.O fluxo ocorre da seguinte forma:
**2**.O cliente cria uma mensagem de requisição.
**3**.A mensagem é serializada usando protobuf.
**4**.O gRPC envia a requisição ao servidor através do HTTP/2.
**5**.O servidor processa a solicitação.
**6**.O servidor retorna uma única mensagem de resposta.
**7**.O cliente recebe a resposta.


### Server-streaming calls

Um RPC de streaming de servidor é semelhante a um RPC unário, exceto que o servidor retorna um fluxo de mensagens em resposta à solicitação de um cliente. Depois de enviar todas as suas mensagens, os detalhes de status do servidor (código de status e mensagem de status opcional) e os metadados de trilha opcional são enviados ao cliente. Isso conclui o processamento no lado do servidor. O cliente conclui assim que tiver todas as mensagens do servidor.

```
Cliente -------> Servidor - 1 requisição

Cliente <------- Servidor - resposta 1

Cliente <------- Servidor - resposta 2

Cliente <------- Servidor - resposta 3

Cliente <------- Servidor - resposta N
```

Exemplo:
[Teste Server Streaming Calls](/pesquisa/server- streaming-calls/)

```
cd server- streaming-calls
python3 -m grpc_tools.protoc \
   -I. \
   --python_out=. \
   --grpc_python_out=. \
   clima.proto

python3 server.py
```

em outro terminal

```
cd server- streaming-calls
python3 client.py
```

### Client-streaming calls

Um RPC de streaming de cliente é semelhante a um RPC unário, mas o cliente envia várias mensagens para o servidor em um fluxo. Quando o cliente termina de enviar os dados, o servidor processa o conjunto recebido e retorna uma única resposta.

Esse modelo é útil quando o cliente precisa enviar uma sequência de eventos, medições ou blocos de dados antes de receber o resultado final.

```
Cliente -------> Servidor - requisição 1

Cliente -------> Servidor - requisição 2

Cliente -------> Servidor - requisição 3

Cliente -------> Servidor - requisição N

Cliente <------- Servidor - resposta única consolidada
```

No exemplo de `client-streaming-calls`, o cliente envia várias temperaturas de uma cidade e o servidor retorna um resumo contendo:

- cidade
- média das temperaturas
- total de medições recebidas

Exemplo:

```
python3 -m grpc_tools.protoc \
   -I. \
   --python_out=. \
   --grpc_python_out=. \
   clima.proto

python3 server.py
```

em outro terminal

```
cd client-streaming-calls
python3 client.py
```

### Bidirectional-streaming calls

Em um RPC de streaming bidirecional, cliente e servidor enviam um fluxo de mensagens ao mesmo tempo. Diferente dos outros modelos, os dois lados podem ler e escrever em paralelo durante toda a chamada.

Esse modelo é ideal para chat em tempo real, telemetria contínua, jogos online e qualquer cenário em que ambos os lados precisam trocar mensagens continuamente.

```
Cliente -------> Servidor - mensagem 1
Cliente <------- Servidor - resposta 1

Cliente -------> Servidor - mensagem 2
Cliente <------- Servidor - resposta 2

Cliente -------> Servidor - mensagem N
Cliente <------- Servidor - resposta N
```

No exemplo de `bidirectional-streaming-calls`, o cliente envia mensagens de chat e o servidor responde em tempo real para cada mensagem recebida.

Exemplo:
[Teste Bidirectional Streaming Calls](/pesquisa/bidirectional-streaming-calls/)

```
python3 -m grpc_tools.protoc \
   -I. \
   --python_out=. \
   --grpc_python_out=. \
   chat.proto

python3 server.py
```

em outro terminal

```
cd bidirectional-streaming-calls
python3 client.py
```



### Comparação de Desempenho: gRPC vs REST API

A fim de demonstrar empiricamente as diferenças entre o gRPC (HTTP/2 + Protobuf) e o REST (HTTP/1.1 + JSON), o projeto foi expandido para suportar ambas as arquiteturas de comunicação interna.

#### Arquitetura de Benchmark (Isolada em Kubernetes)

Conforme a arquitetura solicitada pelo professor, o ambiente de teste roda sobre Kubernetes, onde o **HClient** (cliente de carga com k6) fica fora do cluster e ataca o Gateway (Microsserviço P).
O Gateway roteia a comunicação usando as seguintes rotas idênticas para o HClient, diferindo apenas na topologia interna:

- `GET /api/v1/livros` → O Gateway comunica via **gRPC** com o Microsserviço A.
- `GET /api/v2/livros` → O Gateway comunica via **REST** com o Microsserviço A-REST.

Para garantir que a concorrência não seja um gargalo falso a favor do gRPC, o Microsserviço A-REST foi implementado utilizando **FastAPI** (ASGI/Assíncrono) com `asyncpg`, oferecendo um modelo de concorrência não-bloqueante justo e comparável à pool de threads nativa do gRPC. Além disso, cada protocolo roda no seu próprio pod isolado (`microsservico-a` vs `microsservico-a-rest`), viabilizando a recolha de métricas de CPU/RAM isoladas com `kubectl top pods`.

#### Diferenças Técnicas Evidenciadas

1. **Serialização e Tamanho do Payload**: O gRPC usa Protobuf (binário, compacto, requer contrato `.proto` compilado), enquanto REST usa JSON (texto plano, maiores payloads, mas legível para humanos sem serializador específico).
2. **Protocolo de Transporte**: O gRPC impõe HTTP/2 com multiplexação real e streams bidirecionais numa única conexão TCP. O REST padrão opera frequentemente sobre HTTP/1.1, suscetível ao *Head-of-Line Blocking* e requerendo *connection pooling* pesado.
3. **Semântica**: O gRPC invoca métodos remotos fortemente tipados (RPC). O REST baseia-se em URLs para representação de recursos e verbos HTTP (GET, POST).

#### Como Executar o Benchmark

O benchmark utiliza a ferramenta **k6** para simular múltiplos *Virtual Users* (VUs) simultâneos, garantindo que o teste exerça pressão através da concorrência, cenário onde a multiplexação do HTTP/2 sobressai.

```bash
# Iniciar a infraestrutura (Kubernetes recomendado, ou Docker Compose)
docker compose up -d

# Rodar o script de carga com ramp-up (v1 e v2 sequencialmente)
k6 run benchmark/grpc_vs_rest.js
```
